# AgentBloc v3.1 — Paperclip Emission Target

> Status: **Design proposal**. Not yet implemented.
> Owner: Pablo. Last updated: 2026-05-09.

## Goal

Make AgentBloc emit a [Paperclip-importable company package](https://github.com/paperclipai/paperclip/blob/master/docs/companies/companies-spec.md) (the `agentcompanies/v1` spec) alongside the existing portable spec folder, from the same canonical inputs. The user-facing pitch becomes: "AgentBloc designs your AI company through interview, emits a Paperclip-importable folder, you run `npx companies.sh add ./your-company` and it runs. For the parts Paperclip can't run yet (custom MCPs, n8n flows, webhook receivers), AgentBloc tells you exactly what to build first."

## Strategic framing

The `agentcompanies/v1` spec is **vendor-neutral by design** ("intended to be usable by any agent-company runtime, not only Paperclip"). It extends the existing Agent Skills `SKILL.md` standard upward into company / team / agent composition. Adding it as an emission target is therefore not coupling to Paperclip; it's adopting a portable specification that Paperclip happens to be the reference consumer for.

The complementary positioning relative to Paperclip:

| Paperclip answers | AgentBloc answers |
|---|---|
| "Here's a folder. Run my company." | "What should the folder look like?" |
| Builds skills if you write them | Researches which skills exist vs. need to be built |
| Assumes the org chart | Designs the org chart from a workflow interview |
| Heartbeat-driven runtime | Architect-only; emits a spec the runtime consumes |

Paperclip ships [16 pre-built companies](https://github.com/paperclipai/companies). AgentBloc fills the gap before that point: most users don't fit a pre-built company, and don't know which agents they need.

## Three emission targets

Phase 5 gains a `--target` flag. Default unchanged.

| Target | Output | Audience |
|---|---|---|
| `portable` (default) | Existing `<destination>/` shape per [spec-folder-structure.md](../.claude/skills/agentbloc/references/spec-folder-structure.md) | Codex, Cursor, Gemini, OpenClaw, generic Claude Code |
| `paperclip-v1` | `agentcompanies/v1` package (COMPANY.md root, agents/, skills/, projects/, .paperclip.yaml) | Paperclip operators |
| `both` | Two sibling folders: `<destination>/portable/` + `<destination>/paperclip/` | Users who want optionality, hand-offs across runtimes |

Selection happens at Phase 5 entry. SKILL.md asks once: "Where do you want this spec emitted? (1) Portable folder for any AI coding agent (default), (2) Paperclip-importable company package, (3) Both." User picks; spec-engine takes a `target` arg.

## What changes upstream (Phases 1-4): nothing

The three canonical inputs (`business-graph.json`, `agent-profiles.yaml`, `inventory.yaml`) already carry every field needed to emit either target. No interview changes, no schema bumps to upstream files. The mapping is mechanical, applied at Phase 5.

The only optional addition is a Phase 4 line in the Spec Review walkthrough: "You'll get this team in `paperclip-v1` shape. Here's where each piece lands." That's narrative, not a schema change.

## Canonical mapping (AgentBloc → Paperclip)

This is the contract `spec-engine` enforces when `target=paperclip-v1`.

### Top-level

| AgentBloc | Paperclip | Notes |
|---|---|---|
| `business.type` | `COMPANY.md` frontmatter `description` | Free text |
| `business.owner` | `COMPANY.md` frontmatter `authors[0].name` | Single-author convention |
| `business.size` | `COMPANY.md` body, "About" paragraph | Narrative |
| `processes[].pain` | `COMPANY.md` body, motivation block | Per-process bullets |
| `decision_patterns[]` | `COMPANY.md` body, "Operating rules" section | Free-text rules survive verbatim |
| `security_profile.regimes_activated` | `COMPANY.md` body, "Compliance posture" + `metadata.agentbloc.security_profile` extension | Lossy in base spec; preserved in extension |
| `team.name` | `COMPANY.md` frontmatter `slug` | Stable identity |
| `team.topology` | `metadata.agentbloc.topology` extension | Paperclip only has `reportsTo`; topology preserved as extension |

### Per agent

| AgentBloc field | Paperclip location | Notes |
|---|---|---|
| `agents[].id` | folder name `agents/<slug>/` | kebab-case preserved |
| `agents[].role` | `agents/<slug>/AGENTS.md` frontmatter `title` | One-line title |
| `agents[].goal` | AGENTS.md body, "What you produce" section | Narrative |
| `agents[].backstory` | AGENTS.md body, opening paragraph | Narrative |
| `agents[].tools[]` | AGENTS.md frontmatter `skills[]` | Shortname references resolved against `skills/<shortname>/SKILL.md` |
| `agents[].dependencies[]` | AGENTS.md `reportsTo` (hierarchy only) + body "Who you hand off to" | Pure-mesh teams set `reportsTo: null` on every agent; the dependency edges live in narrative |
| `agents[].triggers[].type=cron` | `.paperclip.yaml` `agents.<slug>` adapter config + `routines.<workflow>.triggers[].kind=schedule` | Cron lives in vendor extension, not base spec |
| `agents[].triggers[].type=inter-agent` | AGENTS.md body, "What triggers you" section | Paperclip handles agent-to-agent via skill invocation; lossy in schema, preserved in narrative |
| `agents[].autonomy` | `metadata.agentbloc.autonomy` extension | Approval gates are runtime-side in Paperclip |
| `agents[].blast_radius` | `metadata.agentbloc.blast_radius` extension | Not in base spec |
| `agents[].escalation` | AGENTS.md body, "Escalation" section | Free-text |
| `agents[].model` | `.paperclip.yaml` `agents.<slug>.adapter.model` (when `adapter.type=claude_local`) | Adapter-specific |

### Per tool / skill

| AgentBloc field | Paperclip location | Notes |
|---|---|---|
| `tools[].tool_id` | folder name `skills/<slug>/` | One skill package per tool |
| `tools[].tier` | `metadata.agentbloc.tier` extension | Not in base spec; Paperclip assumes the skill exists |
| `tools[].tier=EXISTS-MCP` + `evidence.url` + `evidence.last_commit` | `metadata.sources[]` block (`kind: github-file`, `repo`, `commit`, `attribution`, `usage: referenced`) | SHA pinning becomes mandatory for tier-1 tools |
| `tools[].mcp_server.package` + `version` + `installed_via` | `metadata.agentbloc.mcp_server` extension | Detail Paperclip's runtime can ignore or consume |
| `tools[].mcp_server.required_scopes[]` | `COMPANY.md` `requirements.secrets[]` (aggregated across all tools) | Paperclip prompts for these on import |
| `tools[].wrapper.*` (NEEDS-MCP-WRAPPER tier) | SKILL.md body, "How to build this skill" section + `metadata.agentbloc.wrapper.*` extension | Tier-2 skills ship as build instructions, not as runnable code |
| `tools[].n8n_flow.*` / `webhook.*` / `manual.*` | SKILL.md body sections + `metadata.agentbloc.{n8n_flow,webhook,manual}` extensions | Same pattern: narrative + extension preserves the structured data |

### Per workflow / process

| AgentBloc field | Paperclip location | Notes |
|---|---|---|
| `workflows[].id` | folder name `projects/<slug>/` | One project per workflow |
| `workflows[].type` | `PROJECT.md` body, "Pattern" section | (sequential / parallel / loop / event-driven / conversational) |
| `workflows[].agents[]` | `PROJECT.md` body, "Agents" list | Names cross-link to `agents/<slug>/` |
| `workflows[].steps[]` | one `tasks/<n>-<slug>/TASK.md` per step (recurring: true) | Each step becomes a task; project owns the orchestration |
| `workflows[].trigger` (cron) | `.paperclip.yaml` `routines.<project-slug>.triggers[]` | Cron lives in vendor extension |
| `workflows[].why` | PROJECT.md body, "Why this pattern" section | Narrative |

## Lossy items (flagged explicitly in SPEC-EMISSION-REPORT.md)

Three concepts don't survive a round-trip cleanly. The emission report enumerates them under a `paperclip_compatibility_notes` section so users know what they're trading.

1. **Mesh topology**. Paperclip's only org primitive is `reportsTo`. Pure-mesh teams (every agent calls every other) collapse to a flat list. The relationships live in narrative + the `metadata.agentbloc.topology` extension. Re-import via the AgentBloc inputs preserves it.
2. **Inter-agent triggers**. Paperclip routes agent-to-agent through skill invocation, not as a first-class trigger type. The trigger detail lives in narrative. If the runtime needs explicit routing, the build session (post-import) wires it.
3. **Readiness tiering**. Paperclip assumes the skill exists. The tier classification rides in `metadata.agentbloc.tier` on each SKILL.md. A sibling `.agentbloc/inventory.md` (outside the canonical Paperclip package, in the destination root) is also written so the build session can sort skills by readiness when planning the build order.

## Phase 5 protocol changes

The 6-step protocol is preserved. The change is per-step branching on `target`.

```
Step 1 — Resolve destination + target. Snapshot inputs (unchanged).
Step 2 — Emit core docs:
           portable: README/AGENTS/CLAUDE/ROADMAP.md (current behavior)
           paperclip-v1: COMPANY.md + .paperclip.yaml + README.md
           both: emit each into its sibling subfolder
Step 3 — Emit agents:
           portable: agents/<id>/{role,prompts,tools,blast-radius,escalation}.md (5 files)
           paperclip-v1: agents/<slug>/AGENTS.md (1 file with frontmatter + body, content collapsed from the 5 portable files)
Step 4 — Emit integrations / skills:
           portable: integrations/INVENTORY.md + 5 tier subfolders (current behavior)
           paperclip-v1: skills/<slug>/SKILL.md (one per tool, tier-aware body + metadata.sources + metadata.agentbloc.tier)
                         + .agentbloc/inventory.md sibling (NOT inside the canonical Paperclip package; lives at destination root next to it for build-session triage)
Step 5 — Emit workflows / projects:
           portable: workflows/<id>.md (current behavior)
           paperclip-v1: projects/<slug>/PROJECT.md + projects/<slug>/tasks/<n>-<step-slug>/TASK.md per step
                         + .paperclip.yaml routines extension for cron triggers
Step 6 — Emit governance + runtime:
           portable: governance/ + runtime/reference-impl/ (current behavior)
           paperclip-v1: SKIP runtime/reference-impl/ (Paperclip is the runtime via adapters)
                         emit governance/ as advisory siblings (Paperclip enforces approvals via heartbeat / approval gates at runtime)
                         per-agent autonomy + blast_radius rides as metadata.agentbloc.* extensions on AGENTS.md
Step 7 (renumbered — was 6 in the portable-only protocol) — Emit SPEC-EMISSION-REPORT.md with target-aware fields:
         input SHA256s (unchanged)
         per-target file list + byte counts
         paperclip_compatibility_notes (lossy items called out)
         hand-off command:
           portable: "Open <destination>/ in any AI coding agent. Read README.md."
           paperclip-v1: "Run npx companies.sh add <destination>/ in Paperclip"
```

## Validation contract for `target=paperclip-v1`

`spec-engine` adds these checks before writing SPEC-EMISSION-REPORT.md when target includes paperclip-v1:

- Every agent in `agent-profiles.yaml` has a corresponding `agents/<slug>/AGENTS.md`
- Every tool in `inventory.yaml` has a corresponding `skills/<slug>/SKILL.md`
- Every skill referenced in any agent's frontmatter `skills[]` resolves to an actual `skills/<slug>/SKILL.md` file (or to an external pinned source)
- Every cron trigger across agents and workflows is reflected in `.paperclip.yaml` `routines.*` or `agents.*.adapter.*`
- Every required credential (`required_scopes`, `env_secret_name`) is aggregated into COMPANY.md `requirements.secrets[]`
- For tier=EXISTS-MCP tools: the SKILL.md `metadata.sources[]` block includes a SHA-pinned commit (using `evidence.last_commit` on the inventory entry as the resolution date)
- The agentcompanies/v1 schema validation passes (run a structural check against the spec)

Failures emit `SPEC-EMISSION-FAILED-REPORT.md` per the existing protocol (step number, root cause, suggested resolution).

## Implementation phases

Roughly 10-14 CC-hours across three phases.

| Phase | Work | Effort | Output |
|---|---|---|---|
| P1 — Templates + fixtures | Write 11 template files, validate against the worked example | 4 CC-hours | `templates/paperclip-v1/` populated |
| P2 — spec-engine target branching | Add `target` arg, per-step branching, paperclip-v1 validation contract | 5 CC-hours | spec-engine emits both shapes correctly |
| P3 — Phase 5 + SKILL.md wiring | Add target-selection prompt at Phase 5 entry, update phase-5-spec-emission.md and SKILL.md | 2 CC-hours | User-facing target flow works end-to-end |
| P4 — SPEC-EMISSION-REPORT schema bump | Add `paperclip_compatibility_notes` + per-target file list to the report schema | 1 CC-hour | Report surfaces lossy items + hand-off command |

Phase 1 should run first since it grounds the templates in real artifacts. Phases 2-4 are sequential.

## Open questions

1. **Default target**. Should `paperclip-v1` become the default once stable, or stay opt-in? Argument for default: it's the most user-actionable output. Argument against: portable shape is more universal. Proposal: stay opt-in for now; promote after consulting deployments validate the round-trip.
2. **`both` mode duplication**. When emitting both shapes, the canonical inputs are read once but emission walks them twice. Is the file duplication acceptable, or should the two shapes share a single agent / skill source via symlinks / source references? Proposal: duplicate. Symlinks break Windows imports and add complexity for a marginal gain.
3. **Companies catalog submission**. Should AgentBloc-emitted companies be opinionated about submitting back to [paperclipai/companies](https://github.com/paperclipai/companies)? Proposal: emit a sibling `CONTRIBUTING-TO-CATALOG.md` that explains the contribution path, but don't auto-submit.
4. **Reverse direction**. Could AgentBloc consume an existing Paperclip company package and produce a Business Graph from it (Phase 1 reverse)? Out of scope for v3.1; potential v3.2.
