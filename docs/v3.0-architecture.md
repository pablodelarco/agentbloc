# AgentBloc v3.0 ‚Äî Architecture (Spec Engine)

**Status:** Design locked 2026-04-28
**Branch:** `v3.0-spec-engine`
**Predecessor:** v2.0 shipped a design surface with unwired runtime; v2.5
attempted to wire the runtime (12 commits archived locally on `v2.5-runtime`,
preserved as REFERENCE IMPLEMENTATION inside v3.0 output).

## One-line definition

AgentBloc v3.0 is a Claude Code skill that interviews a user about a manual
business workflow, deeply researches which tools (existing MCPs, raw APIs,
n8n flows, custom integrations) can do each step, and emits a portable
**project spec folder** that any AI coding agent ‚Äî Claude Code, Codex, Gemini,
Cursor, OpenClaw ‚Äî can build from without ambiguity.

## The fundamental shift

**v2.0 / v2.5:** AgentBloc tries to be the architect AND the builder. Phase 5
emits running scripts (wake.md, cron lines, hook configs, settings.json patches).

**v3.0:** AgentBloc is the architect only. Phase 5 emits a **build-ready folder
of markdown + YAML + JSON**. A separate AI coding session ‚Äî in any tool ‚Äî does
the implementation using the spec folder as ground truth.

This split lets AgentBloc focus exclusively on the conversational design
problem (which is genuinely hard) and lets implementation tooling stay open
(any AI coding agent in any environment).

## Six phases

```
INTERVIEW  ‚Üí  DESIGN  ‚Üí  DEEP TOOL DISCOVERY  ‚Üí  SPEC REVIEW  ‚Üí  SPEC EMISSION  ‚Üí  SPEC EVOLUTION
  (deep)    (general)      (per-step)             (walkthrough)    (folder out)    (rerun on change)
```

| Phase | v2.5 name | v3.0 name | Status |
|---|---|---|---|
| 1 | Deep Interview | Deep Interview | UNCHANGED |
| 2 | General Design | General Design | UNCHANGED |
| 3 | Deep Integration Analysis | **Deep Tool Discovery** | EXTENDED |
| 4 | Confirmation + Dry Run | **Spec Review** | REFRAMED |
| 5 | Deployment | **Spec Emission** | RENAMED + REWRITTEN |
| 6 | Evolution | **Spec Evolution** | SIMPLIFIED |

### Phase 1 ‚Äî Deep Interview (unchanged)
Unchanged from v2.0/v2.5. Outputs `business-graph.json`. References stay as-is.

### Phase 2 ‚Äî General Design (unchanged)
Unchanged. `designer-agent` subagent emits `agent-profiles.yaml`. References stay as-is.

### Phase 3 ‚Äî Deep Tool Discovery (extended)
This is where v3.0 lives. Today's Phase 3 produces `integration-manifest.yaml`.
v3.0 produces `INVENTORY.md` plus per-tool subfolders, ranking each tool by
**readiness tier**:

| Tier | Meaning | Output |
|---|---|---|
| EXISTS-MCP | Public MCP server exists, install instructions known | `integrations/existing/<tool>.md` |
| NEEDS-MCP-WRAPPER | API exists, no public MCP, wrapper buildable via `mcp-builder` skill | `integrations/needs-mcp-wrapper/<tool>/` with OpenAPI spec + BUILD.md + ENDPOINTS.md |
| NEEDS-N8N-FLOW | Visual / branching logic best done in n8n | `integrations/needs-n8n-flow/<tool>-flow.json` stub |
| NEEDS-WEBHOOK | Event-driven; receiver must be built | `integrations/needs-webhook/<tool>-receiver.md` spec |
| MANUAL | No automation path possible / advisable | `integrations/manual/<tool>.md` documenting why |

Tier assignment is the high-leverage decision. The protocol uses 4 steps:
1. **MCP Search** (existing 4-step protocol from `mcp-integration-protocol.md`)
2. **API Investigation** ‚Äî if no MCP, find OpenAPI spec; assess wrap-ability
3. **n8n Suitability** ‚Äî branching logic, multi-service, polling? n8n wins
4. **Manual Triage** ‚Äî exotic / one-off / human-required

Spawn `browser-discovery` subagent (existing) for unknown services. Spawn
`mcp-builder` skill (referenced) when synthesizing a wrapper.

Sub-gate: `tool_inventory_complete` (every workflow step has a tier
assignment with evidence; no unverified claims).

### Phase 4 ‚Äî Spec Review (reframed)
Walk the user through the proposed spec folder shape (without writing it
yet). Confirm:
- All workflows have falsifiable success criteria
- All agents have unambiguous role/goal/blast-radius
- All tools have a tier + readiness assessment
- All security envelopes (PII, GDPR, audit, kill-switch) are addressed
- Build effort estimate is realistic (~ how many CC-hours / human days)

No "dry run" since nothing executes in v3.0. Spec Review replaces it with
a walkthrough + sign-off ritual.

Sub-gate: `spec_review_signed_off`.

### Phase 5 ‚Äî Spec Emission (renamed + rewritten)
The `spec-engine` subagent (replaces `deploy-engine`) writes the build-ready
folder. Single sub-gate: `spec_folder_emitted`. No `runtime_wired`, no
`monitor_wired`, no DEPLOY-FAILED-REPORT.md cascade.

### Phase 6 ‚Äî Spec Evolution (simplified)
Without a live runtime, Evolution is just "rerun AgentBloc on the spec folder
when requirements change." Reads the existing `spec-folder/` as ground truth,
re-interviews where needed, emits an updated spec. No audit-log forensics.

This is dramatically simpler than v2.5 Phase 6. References that monitored
runtime audit logs are removed.

## Phase gate sub-gates

| Phase | v2.5 sub-gates | v3.0 sub-gates |
|---|---|---|
| 1 | `business_graph_validated` | UNCHANGED |
| 2 | `agent_profiles_validated` | UNCHANGED |
| 3 | `mcp_integrations_verified` | + `tool_inventory_complete` |
| 4 | (none) | `spec_review_signed_off` |
| 5 | `deployment_artifacts_emitted` + `runtime_wired` + `monitor_wired` | `spec_folder_emitted` |
| 6 | (entry preconds on cron/webhook/inter wired) | (entry presupposes spec folder exists) |

Net effect: 3-sub-gate Phase 5 (with two failure cascades) collapses to a
single sub-gate. Phase 6 entry preconditions simplify to "spec folder
exists and validates."

## Build sequence (5 phases of v3.0 work)

| Phase | Content | Effort |
|---|---|---|
| **P1** | Architecture doc (this) + SKILL.md rewrite + memory + branch | ~3h CC |
| **P2** | Reference surgery: rewrite Phase 5/6, add inventory + mcp-synthesis + n8n + webhook spec | ~6h CC |
| **P3** | New `templates/spec-folder/` ‚Äî one template per output file | ~5h CC |
| **P4** | spec-engine subagent (replaces deploy-engine + runtime-engine) | ~3h CC |
| **P5** | Worked example: run end-to-end on arco-rooms; produce reference spec folder | ~4h CC |

Total: ~21h CC over ~10 elapsed days. About **half** the v2.5 budget.

## Ship gate

A Claude Code user can:
1. Activate AgentBloc (`/agentbloc` or trigger phrase)
2. Walk through Phases 1-4 conversationally
3. Receive a `<project>/` folder with the canonical structure
4. Hand the folder to a Claude Code session (or Codex / Gemini / Cursor /
   OpenClaw) and have that session build the team without re-asking
   AgentBloc-level questions

Validation: produce the arco-rooms-property-mgmt example as the worked
spec folder. Open a fresh Claude Code session in that folder, run
`/init` (or paste AGENTS.md), and verify the session can describe the
build plan from the folder alone.

## What's NOT in v3.0

- Live runtime emission (cron, hooks, telegram round-trip) ‚Äî was v2.5 scope
- Inspector dashboard ‚Äî was v2.5 D2; n/a without runtime
- Cost CLI ‚Äî was v2.5 D4; n/a without runtime cost data
- Briefing-agent monitor ‚Äî was v2.0 D-93; documented in spec output as
  optional but not built
- Standalone CLI (`bin/agentbloc`) ‚Äî out of scope for v3.0; v3.1 candidate

## Open questions to lock during build

1. **CLAUDE.md vs AGENTS.md content overlap.** Both ship in spec folder;
   what does each emphasize? AGENTS.md = universal context; CLAUDE.md =
   Claude-Code-specific (skills, subagents, hooks, conventions).
2. **Spec folder destination.** Default = current working directory of the
   AgentBloc session. Override = explicit user-provided path. No default
   to `~/agentbloc-projects/` ‚Äî too opinionated.
3. **Should v3.0 include any non-Claude-Code adapter docs in the repo?**
   No. The spec folder ITSELF is the cross-tool artifact. The skill stays
   pure Claude Code.
