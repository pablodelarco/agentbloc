Arco Rooms is a property management business serving roughly 30 rental properties across multiple owners in Almería, Spain. Pablo is the solo operator. Before this team, end-to-end collection, matching, and reporting consumed 2-3 hours every evening and was error-prone (missed invoices created late fees; mismatched bank transactions caused tenant disputes).

## How it works

The team operates on a sequential nightly pipeline driven by cron. Each agent's output feeds the next:

```
22:00  gestor-documental  →  invoices.json
22:30  gestor-cobros      →  matches.json
23:00  recepcionista      →  Telegram (per-owner)
```

A second event-driven workflow (`unmatched-payment-alert`) wakes recepcionista in real time when gestor-cobros detects 3+ unmatched payments, sending an alert to Pablo's main Telegram thread.

## Operating rules

These rules surfaced from the interview phase. Each one shapes behavior across multiple agents:

1. **Overdue policy**. If an invoice is overdue by more than 7 days, send a formal notice to the tenant. Owned by gestor-cobros at match time.
2. **Reconciliation escalation**. If a bank transaction has no matching invoice after 48 hours, escalate to Pablo for manual review. Owned by gestor-cobros via the `unmatched-payment-alert` workflow.
3. **Portal fallback**. If a provider portal is unreachable for 3 consecutive days, send a Telegram alert and fall back to Gmail scraping for that provider. Owned by gestor-documental.

## Compliance posture

The team handles PII (tenant names, addresses, DNI/NIE) and financial data (invoices, IBANs, transaction descriptions). GDPR applies; redaction rules are enforced before any audit log line.

- DNI/NIE always redacted in logs and inter-owner communications
- IBANs truncated to last 4 digits in logs
- Per-owner Telegram messages contain owner-scoped PII by design (the owner is the lawful recipient of their own tenants' data per the property management contract); cross-owner leaks are blocked by per-owner thread routing
- 12-field JSONL audit trail with append-only retention

## Topology and handoffs

Three peer agents (mesh topology). All have `reportsTo: null` because there is no hub or CEO. The peer-call edges:

- gestor-cobros depends on gestor-documental's `invoices.json` (file-based handoff)
- recepcionista depends on gestor-cobros' `matches.json` (file-based handoff)
- recepcionista may peer-call gestor-cobros via inter-agent message for ad-hoc payment-status queries
- gestor-cobros may peer-call recepcionista via inter-agent message when unmatched payments exceed threshold (event-driven alert workflow)
