### New decisions (autonomous, per PDF + REQUIREMENTS + prior-phase patterns)

#### Autonomy enforcement layer (resolves AUTON-01, AUTON-02)

- **D-84 (Autonomy is enforced at TWO layers: per-agent SKILL.md prose AND a runtime PreToolUse hook keyed on the agent's `autonomy` registry field):** Phase 12 already shipped 3 deployed-agent-skill templates (`-full`, `-semi`, `-supervised`); each contains autonomy-appropriate prose. Phase 14 adds the runtime enforcement seam: a PreToolUse hook (`.claude/hooks/autonomy-gate.sh`) that intercepts every external-side-effect tool invocation (defined as: any MCP tool whose name starts with `send_`, `post_`, `create_`, `update_`, `delete_`, `transfer_`, `pay_`, OR any Bash invocation that writes outside `.agentbloc/agents/<agent-id>/`).

  The hook reads `registry.yaml` to resolve the running agent's autonomy level (`full | semi | supervised`):
  - `full` -> tool proceeds unchanged. PostToolUse hook still logs the call.
  - `semi` -> hook calls `approval-router.sh telegram-request <agent-id> <tool> <args-redacted-summary> <reversibility-tag>`; pauses up to 600s for approval; on approve -> proceed; on deny -> block with `result: blocked, reason: user-denied`; on timeout -> block with `result: blocked, reason: approval-timeout`. Approval log appended to `.claude/agents/logs/<YYYY-MM-DD>/approvals.jsonl` per AUTON-03.
  - `supervised` -> hook ALWAYS pauses; never auto-proceeds; same approval-router path. The difference vs `semi` is that supervised also intercepts internal-side-effect tools (writes to `.agentbloc/agents/<agent-id>/state.json` excluded; writes to `.agentbloc/agents/<other-id>/...` intercepted).

  Why two layers: the SKILL.md prose informs Claude's reasoning + provides graceful degradation if the hook is misconfigured; the hook provides deterministic enforcement that does not depend on the model honoring prose. This matches Phase 12 D-67 narrow-Bash + Phase 13 D-77 three-point kill-switch patterns , defense in depth.

  **Tool classification:** Phase 14 emits `references/autonomy-controller.md` with a definitive table of MCP tool name patterns -> side-effect category (external vs internal vs read-only). Read-only tools (Read, Grep, Glob, fetch-pattern MCP tools) NEVER trigger the hook regardless of autonomy.

  Alternatives considered:
  | Option | Selected |
  |---|---|
  | A. Prose-only enforcement (no runtime hook) | Rejected , relies on model compliance, no audit guarantee |
  | B. Per-tool wrapper subagent that re-injects approval prompt | Rejected , adds latency + token cost per tool call; subagent-spawn pattern was already terminated by Phase 13 D-82 precedent |
  | C. PreToolUse hook + SKILL.md prose two-layer (defense in depth) | ✓ |
  | D. Approval-required-by-default (whitelist instead of blacklist) | Rejected , `full` autonomy semantics require unrestricted action; whitelist would force every agent to be `semi` minimum |

#### Approval router + Telegram thread separation (resolves AUTON-02, AUTON-03, CTRL-01)

- **D-85 (Approval router is a single shell script `approval-router.sh` emitted to `.agentbloc/runtime/approval-router.sh` by deploy-engine; uses Telegram MCP via the `claude` CLI to dispatch + poll; thread-per-agent + dedicated `approvals` thread separate from briefing thread):** Approval requests follow this flow:

  1. PreToolUse hook (D-84) invokes `approval-router.sh telegram-request <agent-id> <tool> <args-summary> <reversibility>`.
  2. Script reads `registry.yaml` for the agent's `escalation_target` (Telegram chat_id) and `approval_thread_id` (separate from `briefing_thread_id` per CTRL-01).
  3. Script POSTs a structured Telegram message to the `approval_thread_id`:
     ```
     [APPROVE/DENY] <agent-id>
     Action: <tool>(<args-summary>)
     Reversibility: <reversibility>
     Reasoning: <agent's stated reason from SKILL.md prose, captured in $TOOL_REASONING env var the hook sets>
     Reply: /approve <correlation_id> | /deny <correlation_id>
     ```
  4. Script polls Telegram for a reply matching `^/approve <correlation_id>$` OR `^/deny <correlation_id>$` for up to 600 seconds (configurable per-agent in registry.yaml `approval_timeout_seconds`). Uses Telegram `getUpdates` long-polling with `offset` to avoid duplicate consumption.
  5. Approval/denial appended to `.claude/agents/logs/<YYYY-MM-DD>/approvals.jsonl` with fields: `correlation_id`, `agent_id`, `tool`, `args_summary`, `reversibility`, `proposal_timestamp`, `decision_timestamp`, `decider_telegram_user_id`, `outcome: approved|denied|timeout`, `reasoning_supplied: <optional reply text after the slash command>`. Per AUTON-03 append-only.
  6. Script exits 0 on approved (hook proceeds), 1 on denied/timeout (hook blocks).

  **Why separate thread:** CTRL-01 mandates the approval queue stays separate from the noisier briefing thread so decisions are not buried. Phase 14 generates BOTH thread IDs at deploy-time (deploy-engine creates them via Telegram API once per team if not already present, persists IDs in registry.yaml + governance.yaml).

  **Slash-command syntax:** `/approve <correlation_id>` not `/approve` because multiple pending approvals may exist simultaneously; correlation_id is the disambiguator. Reply may include freeform reasoning after the command (e.g., `/approve cron-20260501T080000Z-a3f21b yes, this is the May invoice run`); reasoning captured in approvals.jsonl.

  Alternatives considered:
  | Option | Selected |
  |---|---|
  | A. Inline approval in main briefing thread | Rejected , CTRL-01 explicitly demands separation |
  | B. Per-agent dedicated thread (one per agent) | Deferred , v2.0 ships single shared `approvals` thread; per-agent threads belong to v2.5 dashboard scope |
  | C. Reply-by-reaction (👍/👎) instead of slash command | Rejected , reactions are not Telegram-API-pollable as cleanly as message replies; slash commands give explicit text audit |
  | D. Single `approvals` thread + slash command with correlation_id | ✓ |

#### Escalation protocol (resolves AUTON-04, AUTON-05)

- **D-86 (Escalation is a separate Telegram thread `escalations` distinct from `approvals` and `briefing`; messages follow a 4-part template; agent halts and persists `status: error` in last-run.json until a /resume slash command is replied):** When an agent's wake fails (any uncaught exception, any tool returning `result: failure` for a critical action, OR any deliberate `escalate(...)` call from the agent's prose):

  1. Agent appends an entry to its JSONL log: `{action: "escalation", priority: "critical", details: {...}}` , one line.
  2. Agent's wake.md section 6 (state + log write) sets `last-run.json` `status: error` per MEM-04.
  3. Agent invokes `escalation-router.sh telegram-escalate <agent-id> <correlation_id> <what-tried> <why-failed> <options-array> <recommended-next-action>` , a sibling script to approval-router.sh.
  4. Script POSTs to the `escalations_thread_id` (separate from `approvals` per AUTON-04 + D-85; same separation pattern):
     ```
     🚨 ESCALATION , <agent-id>
     Correlation: <correlation_id>
     What I tried: <what-tried>
     Why it failed: <why-failed>
     Options:
       1. <option-1>
       2. <option-2>
       3. <option-3>
     Recommended: <recommended-next-action>
     Reply: /resume <correlation_id> [free-text instructions] | /halt <correlation_id>
     ```
  5. Agent EXITS its wake with `wake_outcome: escalated`. Subsequent wakes (cron / webhook / inter) check last-run.json `status` first; if `error`, the wake template section 1 (after kill-switch check) detects it and short-circuits with `wake_outcome: skipped-prior-error` UNTIL a /resume reply lands.
  6. /resume reply (handled by an inbound Telegram n8n route deploy-engine creates per registry.yaml escalations binding) updates last-run.json `status: idle` and appends free-text instructions to memory.md `Open Items` section (D-64 location). Next cron/webhook fires resumes work.

  **Why the 4-part template:** AUTON-05 explicitly requires "what tried, why failed, options, recommended next action , not just an error stack." This template is the literal contract. Phase 14 emits `references/escalation-protocol.md` with the template + 3 worked examples + the agent-author guidance for what counts as "escalation-worthy".

  **One-shot vs persistent:** Persistence (status=error halts subsequent wakes) is the safer default. The alternative , agent retries on next wake , risks repeated cost spend on a known-broken state. v2.0 ships persistent halt; v2.5 adds opt-in retry budget per agent.

  Alternatives considered:
  | Option | Selected |
  |---|---|
  | A. Escalations to same thread as briefing | Rejected , swamps the daily summary with crisis interruptions |
  | B. Escalations to same thread as approvals | Rejected , conflates "needs decision" with "system broken" semantically |
  | C. Dedicated `escalations` thread + persistent halt | ✓ |
  | D. Auto-retry with exponential backoff before escalation | Deferred to v2.5; v2.0 fails fast |

#### JSONL log schema + path convention (resolves MONITOR-01, MONITOR-02)

- **D-87 (JSONL log entries follow a 12-field canonical schema; one file per agent per day at `.claude/agents/logs/<YYYY-MM-DD>/<agent-id>.jsonl`; append-only; RFC 8785 JCS-canonicalized line-by-line):** Phase 14 emits `references/jsonl-log-schema.md` with this schema as the single source of truth:

  ```json
  {
    "schema_version": 1,
    "timestamp": "2026-04-26T08:00:00.123Z",
    "correlation_id": "cron-20260426T080000Z-a3f21b",
    "agent_id": "gestor-cobros",
    "team": "arco-rooms",
    "action": "tool_call | decision | error | state_change | escalation | approval_request | approval_response",
    "tool": "mcp__plaid__list_transactions",
    "result": "success | failure | skipped | blocked",
    "details": { "amount_count": 14, "errors": [] },
    "duration_ms": 2340,
    "token_count": { "input": 1245, "output": 312, "cached_input": 0 },
    "cost_usd": 0.0231,
    "requires_human": false,
    "priority": "info | warn | critical",
    "locked_by": null
  }
  ```

  - **Required fields (mandatory; emission fails canonicalization gate if missing):** schema_version, timestamp (ms-precision UTC Z), correlation_id, agent_id, team, action (ENUM), result (ENUM), priority (ENUM)
  - **Optional fields (emit when applicable):** tool, details, duration_ms, token_count, cost_usd, requires_human, locked_by
  - **Path:** `.claude/agents/logs/<YYYY-MM-DD>/<agent-id>.jsonl` per REQUIREMENTS.md MONITOR-02 literal (REQUIREMENTS supersedes per D-59 triple-override precedent; the .agentbloc/ alternative path was considered and rejected because logs are git-versionable trails not mutable runtime state)
  - **Append-only:** writes use `>>` redirect. No line edits, ever. Compaction is a future-phase concern (v2.5 per `.agentbloc/runtime/log-rotate.sh`); v2.0 ships uncompacted.
  - **One file per agent per day:** balances grep ergonomics (single file = single rg target for a day's run) vs file-count explosion (one-per-wake would be ~30 files/day per agent). Day boundary is UTC midnight.
  - **Briefing-agent input:** glob `.claude/agents/logs/<TODAY>/*.jsonl` to ingest all agents' day; merge by timestamp ascending; produce daily summary.
  - **Approvals.jsonl + escalations.jsonl:** dedicated files at `.claude/agents/logs/<YYYY-MM-DD>/approvals.jsonl` + `.claude/agents/logs/<YYYY-MM-DD>/escalations.jsonl` use the SAME schema with `agent_id` set to the originating agent + `action: approval_request | approval_response | escalation`.

  Alternatives considered:
  | Option | Selected |
  |---|---|
  | A. Single team log file per day (all agents merged) | Rejected , concurrent writes from cron + webhook agents would interleave at line-boundary risk; per-agent files are append-safe via O_APPEND |
  | B. One file per agent per WEEK | Rejected , file-size grows; daily boundary matches briefing cadence |
  | C. SQLite single DB per team | Deferred to v2.5; introduces dependency contrary to v1.0 file-based decision |
  | D. JSONL per-agent-per-day at REQUIREMENTS path | ✓ |

#### Task locking (resolves CTRL-03)

- **D-89 (Task locks are file-based, single-writer, expiry-bounded; lock files at `.agentbloc/locks/<resource-slug>.lock` containing `{agent_id, correlation_id, acquired_at, expires_at}` JSON):** When an agent works on a shared resource (a bank account, a tenant record, a calendar slot):

  1. Agent atomically attempts: `flock -n .agentbloc/locks/<resource-slug>.lock` then writes the lock content. If `flock` fails (already held), agent reads the existing lock content, checks `expires_at`; if expired (current UTC > expires_at), agent forces release + re-acquires; if not expired, agent sets `wake_outcome: deferred-locked` and exits without further work.
  2. Default `expires_at`: `acquired_at + 30 minutes`. Configurable per-resource in registry.yaml `monitor.lock_defaults`. Agents holding a lock past `expires_at` due to long-running tools are responsible for refreshing (write new `expires_at` mid-operation); this is a documented agent-author concern.
  3. Lock release is explicit: agent writes `null` to the lock file at end of wake (or if escalating). Agent's wake.md section 6 (state + log write) handles release.
  4. JSONL log entry for both `acquire` and `release` actions includes `locked_by` field per MONITOR-01 schema.

  **Resource-slug grammar:** kebab-case, scoped by domain: `bank-bbva-es76-1234`, `tenant-arco-rooms-apartment-3a`, `calendar-google-pablo-monday-9am`. Agents derive these deterministically from registry.yaml `monitor.locked_resources` declarations + the entity-id from their working state.

  **Why file + flock vs Redis / Postgres:** v1.0 file-based-state decision applies. flock is POSIX-standard, atomic, zero-dependency. Per-host scope only , multi-host coordination is out of v2.0 scope (v2.5 web dashboard adds Redis if needed).

  Alternatives considered:
  | Option | Selected |
  |---|---|
  | A. In-memory lock (per-process) | Rejected , agents are separate `claude -p` processes; no shared memory |
  | B. SQLite lock | Deferred , adds dependency for capability flock provides |
  | C. flock + JSON lock file with expiry | ✓ |
  | D. Optimistic locking (compare-and-swap on resource state) | Rejected , requires per-resource versioning logic in every agent |

#### Cost + token tracking (resolves CTRL-02)

- **D-91 (Cost tracking is per-log-line as `cost_usd` + `token_count: {input, output, cached_input}`; aggregated by briefing-agent via summation; references active billing mode from `governance.yaml billing.mode` ENUM `max_subscription | api_key`):** Every JSONL log line emitted by a deployed agent that consumed Claude tokens carries `token_count` and `cost_usd` (zero for non-Claude-token actions like file writes; non-zero for `tool_call` actions where the tool was a Claude API call OR for `decision` actions if the agent's main reasoning ran with metered tokens).

  - **Source:** Phase 12 D-67 narrow-Bash deploy-engine cannot access tokens spent during agent execution directly , Phase 14 emits a wrapper script `.agentbloc/runtime/claude-wrap.sh` that intercepts `claude -p` invocations, reads stdout for token usage from the response trailer (Claude API returns `usage: {input_tokens, output_tokens, cache_read_input_tokens}` per call), maps to `cost_usd` using a static rate table at `references/billing-rates.md` (Sonnet 4.6 rates as of v2.0 ship date), and appends the augmented log line to the agent's JSONL.
  - **Subscription mode:** When `governance.yaml billing.mode = max_subscription`, `cost_usd` is computed AS IF it were API-key mode (so users have a comparable number for budgeting an eventual migration), but the briefing-agent annotates the daily summary with "(included in Claude Max subscription)" footer.
  - **API-key mode:** Real cost; briefing-agent surfaces per-agent + per-team daily and month-to-date totals.
  - **Briefing aggregation:** Sum `cost_usd` across all log lines per agent per day; per team (sum of all agents); month-to-date by reading the last 30 days of `activity-feed.jsonl` files.

  Alternatives considered:
  | Option | Selected |
  |---|---|
  | A. Cost tracked separately (not in log entries) | Rejected , decouples cost from action; harder to attribute |
  | B. Per-tick cost as last-run.json field | Rejected , last-run.json is point-in-time; aggregation requires log line trail anyway |
  | C. Per-log-line `cost_usd` + `token_count` | ✓ |
