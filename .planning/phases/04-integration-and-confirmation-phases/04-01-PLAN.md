**Section content requirements:**

1. **When This Applies:** Claude reads this file when Phase 2 gate is approved and Phase 3 begins. Input is the confirmed agent summary table and per-agent contract cards from design. Also load references/credentials.md and references/prompt-injection.md at this point.

2. **Integration Opening:** Claude presents a brief explanation of what this phase does: "I'll now research the best integration method for every service your agents need. I verify everything with live searches -- no guesswork." Adapt to tech level.

3. **Step 1: Service Inventory:** Extract all unique services from the confirmed agent contract cards (the "Tools" and "Inputs" fields). List them as a checklist. Each service will be analyzed in Steps 2-5. Group by agent to show which agent needs which services.

4. **Step 2: Multi-Method Search Protocol (INTG-01, D-01):** For each service, search integration methods in this strict priority order:
   - Priority 1: Official API -- WebSearch for `{service_name} API documentation`. Record URL, auth method, rate limits.
   - Priority 2: MCP Server -- Search PulseMCP (`list_servers` tool if available, or WebSearch `{service_name} MCP server site:pulsemcp.com OR site:github.com`). Record package name, GitHub stars, last commit, publisher.
   - Priority 3: Playwright browser automation -- WebSearch for `{service_name} login portal`. Note if Playwright MCP can automate the workflow.
   - Priority 4: Email scraping -- Check if the service sends notification emails processable via Gmail MCP (Google Workspace MCP).
   - Priority 5: Webhook interception -- WebSearch for `{service_name} webhook API`. Note if event-driven integration is possible.
   - Priority 6: Manual notification (last resort) -- Telegram notification asking user to handle the action.
   Stop after finding 3 viable options per service (per D-01). Include a note that Arco Rooms example in examples/arco-rooms.md demonstrates this full fallback chain pattern for reference.

5. **Step 3: Evidence Verification (INTG-03, D-03, D-04):** Every integration claim must include:
   - URL (source) -- from WebSearch/WebFetch result or official docs link
   - Package version (if applicable) -- from `npm view` or GitHub release page
   - Last-commit date (for GitHub repos) -- from GitHub repo page or `gh` CLI
   - Publisher info -- npm publisher, GitHub org, or company name
   If any field cannot be found, mark the integration as `[UNVERIFIED]` and warn the user. Include the evidence verification table template from the research (showing Method, Package, Version, Last Commit, Trust, Status columns). State: "An [UNVERIFIED] integration can still be recommended if no better option exists, but you should acknowledge the risk before we proceed."

6. **Step 4: Trust Scoring (INTG-04, D-05, D-06):** Apply the 3-tier trust scoring system:
   - **HIGH:** Official vendor-maintained (Anthropic, Microsoft, Google, Slack, Xero, etc.), OR >500 GitHub stars with active maintenance (commit within 90 days)
   - **MEDIUM:** Community-maintained, 100-500 stars, commit within 180 days, clear documentation
   - **LOW:** <100 stars, >180 days since last commit, unclear maintainer, or no documentation
   Include the full evaluation criteria table (Publisher, GitHub stars, Last commit, Documentation, Known CVEs, Package registry) with thresholds for each tier. State the scoring rule: "Trust level equals the minimum across all criteria. If any single criterion is LOW, overall trust is LOW." Reference the MCP Server Discovery Protocol from CLAUDE.md for consistency (per D-06). For MCP servers handling PII/PHI/financial data, add a mandatory security check: search AgentSeal scores and note known CVEs. Low-trust dependencies get a warning and highlighted alternatives.

7. **Step 5: Decision Matrix Construction (INTG-02, D-02):** One table per service showing recommended, alternative, and fallback methods. Each row includes: method name (with package/tool), trust score, setup complexity (low/medium/high), pros, cons. Include the decision matrix template from the research Code Examples section. Below each table add: Evidence links, Recommendation with rationale, Credential requirement (cross-reference credentials.md per D-13), Prompt injection risk (cross-reference prompt-injection.md per D-14).

8. **Step 6: Security Cross-Reference (D-13, D-14):** Two subsections:
   - **Credential Evaluation (D-13):** For each integration's recommended method, run through the credential decision tree from references/credentials.md. Record: credential type (OAuth/scoped API key/admin), required scope, rotation policy, environment variable naming convention (`AGENTBLOC_{SERVICE}_{TYPE}`).
   - **Prompt Injection Assessment (D-14):** For each agent, determine if it ingests external content through its integrations (emails, web pages, API responses). If yes, assign defense layers from references/prompt-injection.md based on the agent's blast-radius level. Update the prompt injection defense line in the agent's contract card.

9. **Step 7: Integration Presentation and Approval (INTG-05):** Present all decision matrices to the user. For non-technical users, lead with a plain-language summary of what was found for each service before showing the table. For developers, lead with the technical table. Ask: "Review the integration analysis above. For each service, I've recommended the best option. Would you like to change any selection, or shall we proceed?" Wait for explicit approval. The user can change any integration selection at this point.

10. **Integration Gate:** Phase 3 gate artifact is the complete set of integration-enhanced agent contract cards. Each contract card from design now has a "Selected Integrations" section added (showing chosen method, trust score, credential requirement per service). The agent summary table is also updated with integration data. State bar changes to: "Phase 3: Deep Integration Analysis | Gate: approved | Level: {level}" only after user confirms.

11. **Quick Reference:** A compact summary table of the entire protocol steps, trust scoring tiers, evidence requirements, and the multi-method priority order.

**Writing style:** Follow the established pattern from phase-2-design.md -- prescriptive ("do this, then this"), not descriptive ("consider doing this"). Use second person addressing Claude ("You MUST search...", "Present the decision matrix..."). Include concrete examples using the Arco Rooms pattern where appropriate. Include behavior-by-level notes for presenting findings to different tech levels.
