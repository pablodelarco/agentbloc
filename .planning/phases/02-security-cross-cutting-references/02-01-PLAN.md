2. **Scoring Levels** (4-level table with concrete criteria):
   | Level | Name | Tool Access | Data Access | External Comms | Approval |
   | 1 | read-only | Read, Grep, Glob, WebFetch | Read-only access to state files | None | Not required |
   | 2 | write-scoped | Read + Write to specific paths | Read/write to designated state files | None | Not required |
   | 3 | write-unrestricted | Read + Write + Bash (restricted) | Read/write to any state file | None | Required |
   | 4 | send-external | All tools + MCP servers with external side effects | Full data access | Sends emails, messages, API calls | Required |

3. **Scoring Decision Tree** (how Claude assigns a level during Design):
   - Does the agent send data externally (email, Telegram, API POST to third party)? YES -> Level 4
   - Does the agent write to files or databases without path restrictions? YES -> Level 3
   - Does the agent write to specific, pre-defined state files only? YES -> Level 2
   - Agent only reads data? -> Level 1

4. **Approval Matrix**:
   - Level 1-2: `requires_approval: false` -- agent runs autonomously
   - Level 3-4: `requires_approval: true` -- agent pauses before side effects, sends approval request via Telegram, waits for human confirmation
   - Note: During dry run (Phase 4), ALL agents regardless of level have side effects stubbed

5. **Permission Minimization Checklist** (questions Claude asks per agent during Design):
   - Can this agent do its job with read-only access? If yes, Level 1.
   - Can write access be scoped to specific files/paths? If yes, Level 2 instead of 3.
   - Can external sends be batched into a single reporting agent? If yes, move sends to the reporter (Level 4) and keep this agent at Level 1-2.
   - Can the agent use a scoped API key instead of an admin token? If yes, reduce blast radius.
   - Does the agent need Bash access? If not, exclude Bash from allowed tools.

6. **Artifact Template** (agent.yaml blast-radius block):
