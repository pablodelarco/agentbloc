### Task 1: declined-agents-schema.md

Schema reference structured like business-graph-schema.md (Phase 8) + integration-manifest-schema.md (Phase 10):

```markdown
# Declined Agents Schema

> Schema reference for `.agentbloc/graph/declined.json`. ANTIC-04 source-of-truth. Designer subagent reads this file at invocation start to filter anticipation-heuristics.md proposals.

## Table of Contents

- When This Applies
- Schema Definition
- Field Obligation Matrix
- Append-Only Discipline
- Designer Integration Protocol
- Re-introduction Behavior

## When This Applies

Designer Agent reads `.agentbloc/graph/declined.json` after the 5 mandatory schema reads (per <anticipation_pass> block in designer-agent.md). Any anticipated agent proposed by anticipation-heuristics.md whose `agent_id` matches an entry in this file is filtered out before the rendered TABLE returns to the main session.

## Schema Definition

\`\`\`json
[
  {
    "agent_id": "string",
    "business_type": "string",
    "declined_at": "ISO-8601 timestamp",
    "reason": "string | null",
    "correlation_id": "string"
  }
]
\`\`\`

## Field Obligation Matrix

| Field | Tier | Description |
|---|---|---|
| agent_id | REQUIRED | kebab-case ID matching the anticipation-heuristics.md proposed agent |
| business_type | REQUIRED | Business Graph business.type at decline time |
| declined_at | REQUIRED | ISO-8601 UTC timestamp of decline |
| reason | RECOMMENDED | 1-line free-text rationale (not user-facing in v2.0; for the user's own future reference) |
| correlation_id | REQUIRED | Designer invocation correlation ID per Phase 13 D-75 format |

## Append-Only Discipline

Per Phase 14 D-87 trace-integrity pattern. New declines append to the array. Designer never auto-removes entries. User can manually edit the file to UN-decline (delete entries) if they change their mind; the file is plain JSON for that reason.

## Designer Integration Protocol

1. Designer reads `.agentbloc/graph/declined.json` at invocation start (after the 5 mandatory schema reads).
2. If the file does not exist, Designer treats the decline set as empty and proceeds.
3. After requested-agent emission completes, Designer queries anticipation-heuristics.md for `business.type`. For every anticipated agent the heuristics map proposes, Designer checks if `agent_id` appears in declined.json with matching `business_type`. Matches are filtered out before the rendered TABLE.
4. When the user declines an anticipated agent during the Phase 2 review (e.g., "drop the incident tracker"), the main session calls Designer with a structured decline patch; Designer appends to declined.json + removes from agent-profiles.yaml + re-renders the TABLE.

## Re-introduction Behavior

If the user later changes their mind ("actually add the incident tracker after all"), the main session can either:
- Manually edit declined.json to remove the entry, then re-run Designer (which will re-propose from the heuristics map)
- Use a conversational edit ("add an incident tracker") which adds the agent directly without going through the heuristics-map filter (the agent emits with `anticipated: false` because the user added it explicitly)

The schema is intentionally simple. v2.0 ships read-mostly behavior; v2.5 may add a CLI command for declined-management.
```
