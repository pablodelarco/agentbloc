**Step 1: Agent Identification (per DESG-01):**
- One-responsibility-per-agent rule: "Each distinct responsibility in the workflow becomes one agent. If an agent has two unrelated jobs, split it."
- Naming conventions: descriptive role-based names (e.g., "Invoice Collector", "Payment Matcher", "Report Sender"). Avoid generic names ("Agent 1", "Data Handler").
- CrewAI-inspired pattern (per DESG-07): Reference references/frameworks.md for the role-based definition pattern. Map CrewAI's role/goal/backstory to AgentBloc's role/responsibility/contract.
- Identification process:
  1. List all distinct actions from the interview workflow
  2. Group related actions that share context and data
  3. Each group becomes one agent
  4. Name each agent by its primary function
  5. Verify: no agent does two unrelated things; no two agents do the same thing
- Example using Arco Rooms: "Invoice Collector (collects from 6 providers), Payment Matcher (matches bank transactions to invoices), Report Sender (daily Telegram summary)"

**Step 2: Topology Selection (per DESG-02, D-07):**
- Decision tree (adapted from LangGraph patterns, per DESG-07):

  ```
  1. Count the agents identified in Step 1
  2. If 1-3 agents with sequential flow: Pipeline
  3. If 3-5 agents with clear coordinator: Hierarchy
  4. If 3-8 agents iterating on shared artifacts: Mesh
  5. If agents explore unknown paths independently: Swarm
  6. Default: Pipeline (simplest, most predictable)
  ```

- Four topology definitions with brief descriptions and when-to-use criteria:

  | Topology | Pattern | When to Use | Agent Count Sweet Spot |
  |----------|---------|-------------|----------------------|
  | Pipeline | A -> B -> C | Sequential stages, clear handoff points, content/data processing | 1-3 agents |
  | Hierarchy | Coordinator delegates to workers | Centralized coordination needed, tasks have clear boundaries | 3-5+ agents |
  | Mesh | Peer-to-peer, shared artifacts | Iterative refinement, tight feedback loops, collaborative work | 3-8 agents |
  | Swarm | Autonomous exploration | Unknown optimal paths, large-scale parallel collection, discovery tasks | 5+ agents |

- Recommendation with rationale (per D-07): "Recommend ONE topology with a brief explanation: 'Pipeline, because your workflow has [N] sequential stages with clear handoff points between [stage names].' The user can override."
- User override protocol: "If the user disagrees with the topology recommendation, accept the override. Note the reason. Adjust agent connections in the diagram accordingly."
- n8n mental model reference (per DESG-07): "For non-technical users, explain the topology as a visual flow: each agent is a step in the process, and arrows show what data passes between them. Reference frameworks.md for the visual DAG mental model."

**Step 6: Blast-Radius Scoring (per DESG-06, D-09):**
- Auto-scoring protocol:
  1. For each agent, read its contract card (Step 3)
  2. Apply the 4-step scoring decision tree from references/blast-radius.md:
     - What tools does this agent use? (Read-only = Level 1 ceiling)
     - Does it write files? (Write to specific paths = Level 2)
     - Does it write to arbitrary paths or run Bash? (Level 3)
     - Does it send external messages, emails, or API calls with side effects? (Level 4)
  3. Assign the level based on maximum capability, not typical behavior
  4. Update the Blast Radius field in the contract card

- User override mechanism (per D-09): "Present the auto-scored blast-radius in each agent's contract card. The user can override: 'This agent should be read-only, not write-scoped.' Accept the override and adjust the contract card accordingly. Overrides that INCREASE blast-radius are accepted. Overrides that DECREASE blast-radius trigger a warning: 'Reducing this agent to Level 1 means it cannot write state files. Are you sure?'"

- Permission minimization pass: After all agents are scored, run through the 5-item permission minimization checklist from references/blast-radius.md:
  1. Can any Level 4 agent be split into a processor (Level 2) + sender (Level 4)?
  2. Can any agent's write scope be narrowed to specific paths?
  3. Are there agents with Bash access that only need Read/Write?
  4. Can any external API call be replaced with a read-only check?
  5. Are approval gates set for all Level 3+ agents?

- Present the result as an updated agent summary table with blast-radius column.
