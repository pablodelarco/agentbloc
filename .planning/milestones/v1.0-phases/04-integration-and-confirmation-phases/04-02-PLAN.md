**Section content requirements:**

1. **When This Applies:** Claude reads this file when Phase 3 gate is approved and Phase 4 begins. Input is the integration-enhanced agent contract cards (with Selected Integrations, Credential Summary, and Prompt Injection Defense sections added during Phase 3). Also load references/blast-radius.md, references/audit-logging.md, and references/incident-response.md for governance context.

2. **Confirmation Opening:** Claude explains: "Now I'll walk through each agent one at a time. For each one, you'll see exactly what it does, what tools it uses, how it handles failures, and what credentials it needs. You can adjust anything before we test it." Adapt to tech level.

3. **Step 1: Enhanced Contract Card Format (CONF-01, D-07):** Include the complete enhanced contract card template from the research (Pattern 6). The card reuses the design phase contract card format from references/phase-2-design.md but adds three sections:
   - **Selected Integrations** table: Service, Method (from decision matrix), Trust score, Credential type, Setup complexity.
   - **Prompt Injection Defense:** Updated defense layer assignment from integration analysis.
   - **Credential Summary** table: Service, Credential Type, Scope, Rotation period, Env Variable name (following `AGENTBLOC_{SERVICE}_{TYPE}` naming convention).
   Include the full Arco Rooms "Invoice Collector" example from the research Code Examples section as a concrete illustration.
   Include behavior-by-level adaptation: for non-technical users, present a plain-language summary BEFORE the full card ("This agent collects invoices from Xero, Endesa, and Gmail every day at 10pm. It needs your Xero login, Endesa portal password, and Google account access."). For developers, show the full card immediately.

4. **Step 2: Sequential Agent Approval (CONF-02, D-08):** Present agents strictly one at a time in their pipeline/topology execution order. After presenting an agent card:
   - Ask: "Do you approve this agent, or would you like to make changes?"
   - If approved: Move to the next agent. Record approval.
   - If changes requested: Update the card based on feedback (change integration method, adjust blast-radius, modify failure handling, add/remove integrations, change credential type). Present the updated card for re-approval. Changes must propagate: if changing Agent A's output format, check if downstream agents need adjustment.
   - Never present the next agent until the current one is approved.
   Include the "confirmation fatigue" mitigation from research Pitfalls: if the user seems to be rubber-stamping ("just approve them all"), slow down and highlight what's different about each agent vs the previous design. Non-technical users get a "key changes from design phase" highlight before each card.

5. **Step 3: Integration Summary Gate (D-09):** After all agents are individually confirmed, present a final integration summary table. This is the Phase 4 confirmation gate artifact. Format:

   ```markdown
   ## Team Integration Summary

   | Agent | Services | Methods | Trust | Credentials Required |
   |-------|----------|---------|-------|---------------------|
   | [name] | [list] | [list] | [min trust] | [count] |

   **Total services:** {N}
   **Total credentials needed:** {N}
   **Agents requiring human approval:** {list of Level 3-4 agents}
   ```

   Ask: "This is the complete picture of your agent team with all integrations. Before we run the test, does everything look correct?"

6. **Step 4: Dry Run Configuration (CONF-03, D-10):** Configure the mandatory dry run:
   - Ask the user how many real records to process per agent. Default: 5. Claude should suggest a count based on workflow complexity: simple workflows (2-3 services) need 3-5 records; multi-provider workflows (5+ services) need 5-10 records.
   - Explain what dry run means at the user's tech level:
     - Non-technical: "I'll run each agent against your real data, but any action that would change something -- sending a message, writing a file, updating a system -- will be simulated. Nothing gets modified. Think of it as a dress rehearsal."
     - Technical/developer: "All READ operations execute against real data. All WRITE/SEND operations are stubbed: they log what would have happened and return simulated success. Enforcement is dual-layer: prompt-level instruction + PreToolUse hook."
   - Explain the dual-layer enforcement mechanism (per D-11 and research Pattern 7):
     - **Layer 1 (prompt-level):** Agent skill files include a "DRY RUN MODE ACTIVE" section that instructs the agent to log all write/send operations without executing them. Log format: `[DRY RUN] Tool: {tool_name} | Target: {target} | Would have: {action description}`.
     - **Layer 2 (PreToolUse hook):** A bash hook script (`.agentbloc/hooks/dry-run-enforcer.sh`) runs on every tool call. It checks for a `.agentbloc/DRY_RUN_ACTIVE` flag file. When active, it blocks write/send MCP tools by returning exit code 0 with JSON: `{"hookSpecificOutput":{"hookEventName":"PreToolUse","permissionDecision":"deny","permissionDecisionReason":"DRY RUN: {tool} blocked"}}`. Include the CORRECT implementation from the research (exit 0 + JSON, NOT exit code 2). Document the anti-pattern: "Exit code 2 means 'hook crashed,' not 'policy denied.' Always use exit 0 with the JSON deny structure."
     - **Layer 3 (subagent tool restriction):** During dry run, agent subagent definitions use the `tools` field to exclude write/send MCP tools entirely. Include example YAML.
   - Create the `.agentbloc/DRY_RUN_ACTIVE` flag file before starting.

7. **Step 5: Dry Run Execution:** Execute each agent in pipeline/topology order against the configured number of real records. For each agent:
   - Load the agent's skill file with DRY RUN MODE preamble.
   - Run against real data (reads proceed normally).
   - Log every stubbed write/send operation with the `[DRY RUN]` prefix.
   - Record results: operations performed, data read, actions that would have been taken, any errors.
   - If an agent fails (error, missing data, credential issue): record the failure and continue to the next agent. Do not halt the entire dry run.
   State that after all agents complete, the DRY_RUN_ACTIVE flag is removed.

8. **Step 6: Dry Run Report (CONF-04, D-12):** Generate a structured markdown report. Include the complete report template from the research Code Examples section. Format:

   ```markdown
   # Dry Run Report
   **Team:** {name} | **Date:** {date} | **Records:** {N} per agent | **Mode:** DRY RUN

   ## Per-Agent Results
   ### Agent: [Name]
   **Status:** PASS / FAIL / PASS (with warnings)
   | # | Operation | Type | Target | Result |
   [rows for each operation]
   **Errors:** [list or None]
   **Warnings:** [list or None]
   **Verdict:** PASS / FAIL with explanation

   ## Summary
   | Agent | Read Ops | Write Ops (stubbed) | Send Ops (stubbed) | Errors | Verdict |
   [summary row per agent]
   **Overall:** [summary verdict]
   ```

   Include the Arco Rooms 3-agent example from the research as a concrete illustration of a complete dry run report.

9. **Step 7: Final Approval Gate (CONF-05):** Present the dry run report and ask for explicit approval:
   - If all agents passed: "All agents completed the dry run successfully. Review the results above. Approve to proceed to deployment, or request changes."
   - If any agent failed: "Agent {name} encountered issues during the dry run. I recommend addressing these before deployment: [specific issues]. Would you like to adjust the agent and re-run, or proceed with the known issues?"
   - If user approves: State bar changes to "Phase 4: Step-by-Step Confirmation + Dry Run | Gate: approved | Level: {level}". Proceed to Phase 5.
   - If user requests changes: Return to the relevant step (re-confirm an agent, adjust integrations, or re-run the dry run). This may loop back to Step 2 for individual agent changes.

10. **Quick Reference:** Compact summary of the protocol steps, enhanced card format fields, dry run enforcement layers, and the report verdict criteria.

**Writing style:** Same as Plan 01 -- prescriptive, second person addressing Claude. Include concrete Arco Rooms examples. Include behavior-by-level notes. Follow the pattern from phase-2-design.md.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Agent skill file to runtime behavior | Prompt-level dry run instructions could be ignored by the LLM; trust boundary between instruction and execution |
| Dry run flag file to hook enforcement | The .agentbloc/DRY_RUN_ACTIVE flag file controls enforcement; if not created or prematurely removed, enforcement fails |
| Dry run report to user decision | User trusts dry run report accuracy to make deployment decision |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-05 | Tampering | Dry run flag file | mitigate | Document that DRY_RUN_ACTIVE must be created before and removed after dry run; hook script checks for file existence before allowing tool calls |
| T-04-06 | Repudiation | Agent approvals | mitigate | Protocol records each agent approval in conversation history; integration summary gate serves as audit artifact |
| T-04-07 | Elevation of Privilege | Dry run bypassing enforcement | mitigate | Dual-layer enforcement: prompt-level + PreToolUse hook + subagent tool restriction; three independent layers must all fail for bypass (Step 4) |
| T-04-08 | Denial of Service | Dry run consuming excessive tokens | accept | User configures record count; Claude suggests appropriate count based on complexity; default 5 is conservative |
</threat_model>
