**Interview Opening Protocol section:**
- Soft framing instruction (per D-04): Claude says "I'll ask about 15-25 questions across 9 areas to fully understand your workflow." at interview start. No further progress tracking during the interview.
- Technical level detection: reference to SKILL.md technical level assessment (non-technical/technical-basics/developer)

**Progressive Data Classification Protocol section (per D-08):**
- Instruction: "Throughout every category below, apply the auto-detection rules from references/data-classification.md to every user answer. When any data signal is detected (a name = PII, a payment = financial, a diagnosis = PHI), immediately classify it and add to a running security tally. This tally is internal; do not announce each classification to the user."
- Running tally format: internal markdown table tracking data class, signal detected, category where detected
- Cross-reference: "See references/data-classification.md for the complete keyword lists (EN/ES) and confidence thresholds"

**9 Category Sections** (each following this exact format):

For each of the 9 categories (The Problem, The Current Workflow, The Services and Tools, The Data, Data Classification, The People, Edge Cases and Failures, Reporting and Communication, Budget and Constraints):

1. **Seed Questions** (per D-01): 2-3 open-ended mandatory questions. Target 2 per category, 3 for Problem/Workflow/Services which are richer. Total across all 9 categories should be 18-22 questions to stay within the 15-25 soft framing (leaving room for adaptive follow-ups).

   Concrete seed questions per category:

   **The Problem:**
   1. "What's the business problem you want to solve? Describe the pain point in your own words."
   2. "What happens today when this problem isn't handled? What's the cost of the current situation?"
   3. "How will you know the solution is working? What does success look like?"

   **The Current Workflow:**
   1. "Walk me through the current process step by step. What triggers it, and what's the end result?"
   2. "Which steps are manual and repetitive? Where do you spend the most time?"
   3. "How often does this workflow run? Daily, weekly, on-demand?"

   **The Services and Tools:**
   1. "What software, platforms, or tools are involved in this workflow?"
   2. "For each tool, do you have admin access or API access, or do you just use the web interface?"

   **The Data:**
   1. "What data flows through this process? Where does it come from and where does it end up?"
   2. "What format is the data in? Spreadsheets, emails, database records, PDFs?"

   **Data Classification:**
   1. "Does your workflow handle personal information like names, emails, phone numbers, or ID numbers?"
   2. "Are there any payments, invoices, or financial records involved?"

   **The People:**
   1. "Who else is involved in this workflow? Who provides input, who approves things, who receives the output?"
   2. "Who should be notified when something goes wrong or needs manual attention?"

   **Edge Cases and Failures:**
   1. "What goes wrong most often? What are the common exceptions or edge cases?"
   2. "When something fails, what do you do today? Is there a fallback process?"

   **Reporting and Communication:**
   1. "What reports or summaries does this workflow need to produce? For whom?"
   2. "How do you prefer to receive updates: email, messaging app, spreadsheet, dashboard?"

   **Budget and Constraints:**
   1. "Are there budget constraints for the tools and services involved?"
   2. "Are there any technical constraints, compliance requirements, or deadlines I should know about?"

2. **Must-Know Checklist** (per D-02): 3-5 items per category. Internal only, never shown to user. Claude tracks these silently. When all items are covered (through any question path, not necessarily through the seed questions), the category is done. Format as unchecked markdown checkboxes with clear, verifiable items.

   Concrete must-know items per category:

   **The Problem:** core pain point identified, current cost/impact quantified or described, desired outcome articulated, success criteria defined

   **The Current Workflow:** end-to-end process mapped with trigger and output, manual/repetitive steps identified, frequency/volume known, time spent per cycle estimated

   **The Services and Tools:** all software/platforms listed, access level known per tool (admin/API/web-only), integration capability assessed (API available? MCP exists?)

   **The Data:** data types and formats identified, data sources and destinations mapped, volume/frequency of data known

   **Data Classification:** PII presence confirmed or ruled out, financial data presence confirmed or ruled out, PHI presence confirmed or ruled out, data residency/jurisdiction noted (EU triggers GDPR)

   **The People:** all roles identified (operator, approver, recipient), approval chains mapped, notification preferences captured

   **Edge Cases and Failures:** top 3 failure modes identified, current fallback process documented, acceptable failure rate or SLA discussed

   **Reporting and Communication:** report types and recipients defined, delivery channel chosen, frequency determined

   **Budget and Constraints:** budget range or constraints noted, timeline requirements captured, technical/compliance constraints logged

3. **Adaptive Branching** hints: 2-3 conditional follow-up rules per category. Format: "If user mentions X, probe for Y." These guide Claude's adaptive questioning after seed questions.

4. **Data Classification Scan** reminder (per D-08): At the bottom of EVERY category section, include: "**Data Classification Scan:** Apply auto-detection rules from references/data-classification.md to every answer in this category. If any data signal is detected, add to the running classification tally."

**Interview Completion Gate section (per INTV-03):**
- Master completion checklist combining all must-know items into a single internal verification
- Instruction: "Before generating the summary, verify that ALL must-know items across all 9 categories have been covered. If any gaps remain, ask targeted follow-up questions for the missing items only. Do NOT proceed to the summary until all items are checked."

**Summary-of-Understanding Template section (per INTV-04, D-10):**
- Full template with sections: Your Business, The Problem, Current Workflow, Services and Integrations, Data Model, People Involved, Edge Cases, Reporting Requirements, Budget and Constraints, Security Profile
- Security Profile subsection (per D-10): data classes detected, compliance regimes activated (GDPR/HIPAA/PCI), implications for design phase
- Gate instruction: "**Does this accurately capture your workflow? I need your confirmation before proceeding to the design phase.**"
- Instruction: the user must explicitly confirm ("yes", "approved", "ok", "adelante") before Phase 2 begins

**Quick Reference section:**
- Table summarizing: category name, seed question count, must-know item count
- Total question estimate: ~20 seeds + adaptive follow-ups = 15-25 total (aligning with D-04 soft framing)

**Size target:** 200-300 lines. Do not exceed 350 lines. Be concise. Seed questions are single sentences. Checklist items are single phrases. Adaptive branching is 1 line per rule.

**Style:** Follow the pragmatic format established by Phase 2 security reference files. Matter-of-fact tone. No enterprise language. Use markdown tables for structured data. Use heading hierarchy consistently.

**Anti-patterns to avoid:**
- Never show the must-know checklist to the user (D-02)
- Never ask multiple questions per turn (D-03)
- Never limit data classification to the Data category only (D-08)
- Never put progress tracking after the opening framing (D-04)
- Never use "--" (em dash) in text (CLAUDE.md convention, use ":" or restructure)
