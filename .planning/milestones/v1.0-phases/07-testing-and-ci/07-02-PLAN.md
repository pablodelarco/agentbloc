Create `tests/run-tests.sh` -- a self-contained bash test runner per D-07, D-08, D-09.

**Script structure (per RESEARCH.md Pattern 3 -- modular functions):**

```
#!/usr/bin/env bash
set -euo pipefail

# Directory resolution
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
REPO_ROOT="$(cd "$SCRIPT_DIR/.." && pwd)"
SCENARIOS_DIR="$SCRIPT_DIR/scenarios"

# TAP state variables
PASS=0
FAIL=0
TEST_NUM=0

# TAP helper functions
tap_ok() { ... }
tap_not_ok() { ... }

# Validation functions (one per category)
validate_json() { ... }
validate_fields() { ... }
validate_sequence() { ... }
validate_assertions() { ... }
validate_references() { ... }

# Main execution
echo "TAP version 13"
# Run all validations, emit plan at end
```

**Validation category 1: Structural (validate_json)**
- Read each line of the JSONL file
- Pass it through `jq .` to verify valid JSON
- Report one TAP test point per scenario file: "ok N - {filename}: all M lines are valid JSON" or "not ok N - {filename}: X of M lines have invalid JSON"

**Validation category 2: Required fields (validate_fields)**
- For each line, extract `role` via `jq -r '.role // empty'`
- For role "user" or "assistant": verify `content`, `phase`, `gate` fields exist (per D-01)
- For role "assertion": verify `pattern`, `context` fields exist (per D-06)
- For unknown role values: count as error
- Report one TAP test point per scenario file

**Validation category 3: Phase sequence (validate_sequence)**
- Extract all `phase` values from non-assertion lines: `jq -r 'select(.role != "assertion") | .phase' < file`
- Verify phases appear in non-decreasing order (1, 1, 1, 2, 2, 3, ... 6)
- Verify at least phases 1 through 6 are present
- Verify gate transitions: within a phase, gate should go from "pending" to "approved" before next phase starts
- Report one TAP test point per scenario file

**Validation category 4: Assertion matching (validate_assertions)**
- For each assertion line (`role: "assertion"`), extract the `pattern` regex
- Find the most recent preceding `role: "assistant"` line's `content`
- Test if the regex pattern matches the assistant content using `echo "$content" | grep -qP "$pattern"` (or `grep -qE` if PCRE not available; use `grep -qE` for portability since GitHub Actions ubuntu-latest has GNU grep with -E)
- If match: pass. If no match: fail with diagnostic showing expected pattern and the context field
- Report one TAP test point per assertion line (each assertion is its own test)

**Validation category 5: SKILL.md reference links (validate_references per D-09)**
- Parse SKILL.md for all relative file paths matching `references/*.md`
- For each path found, check if the file exists relative to REPO_ROOT
- Also check that `examples/*.md` files referenced in SKILL.md exist
- Report one TAP test point per reference: "ok N - SKILL.md ref: references/phase-1-interview.md exists"

**TAP output requirements per D-08:**
- First line: `TAP version 13`
- Emit `ok N - description` or `not ok N - description` for each test
- After `not ok`, emit `# Expected: ...` and `# Got: ...` as diagnostic lines
- At the very end, emit the plan line: `1..N` (total test count)
- Exit 0 if FAIL is 0, exit 1 otherwise

**Make the script executable:** `chmod +x tests/run-tests.sh`

**Critical implementation notes:**
- Use `grep -qE` (extended regex), NOT `grep -qP` (Perl regex) for portability. Ubuntu-latest and macOS both support `-E`
- Handle empty scenarios directory gracefully: if no .jsonl files found, report a single "not ok - no scenario files found"
- Use `while IFS= read -r line` for reading JSONL lines to preserve whitespace
- Store the "last assistant content" in a variable that updates as you iterate through lines, so assertions always check against the correct preceding turn
- Quote all variables to prevent word splitting on content with spaces

**DO NOT:**
- Require npm, node, python, or any non-standard dependency (per D-07: shell-based, no npm/node)
- Use `jq` features beyond basic parsing (select, -r, -e, // empty) -- keep it simple
- Hard-code scenario filenames -- iterate over all *.jsonl files in the scenarios directory
- Use `set -e` in a way that causes the script to exit on the first test failure (trap individual failures, do not let them kill the script)

**Note on set -e:** The script uses `set -euo pipefail` at the top, but validation functions must handle failures internally (e.g., `if ! cmd; then ...` or `cmd || true` patterns) so individual test failures do not exit the script. Only the final exit code should reflect pass/fail status. Consider using `set +e` before the test loop and `set -e` only for setup, OR structure all validation calls to never produce a nonzero exit in the main flow.
