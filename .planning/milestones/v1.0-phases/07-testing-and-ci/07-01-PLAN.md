  <action>
Create the directory `tests/scenarios/` and the first scenario file `tests/scenarios/arco-rooms.jsonl`.

This scenario represents: Pipeline topology, non-technical user, PII + financial data, GDPR activated, 3 agents (Invoice Collector, Payment Matcher, Report Sender).

**JSONL format per D-01:**
- Every line is a single valid JSON object (no trailing commas, no multi-line)
- User/assistant turns: `role`, `content`, `phase` (1-6 integer), `gate` ("pending"/"approved"/"blocked"), optional `metadata`
- Assertion lines per D-06: `role: "assertion"`, `pattern` (regex), `context` (description)

**Content derivation per D-03 and D-05:**
- Derive turns from `examples/arco-rooms.md`, covering all 6 phases
- Target 60-80 turns (within the 50-100 range per D-03)
- Include key decision points, not full verbatim transcript
- At each phase boundary, the user turn that approves the gate must include `"gate": "approved"` and `"metadata": {"gate_transition": true}`

**Phase structure to cover:**
1. Phase 1 (Interview): 2-3 user questions covering problem, services, data classification. Assistant responds with state bar `**Phase 1: Deep Interview | Gate: pending | Level: non-technical**`. Include assertion validating state bar format, data classification mention (PII, financial, GDPR), and tech level detection.
2. Phase 2 (Design): Assistant presents pipeline topology, 3-agent table with blast radius, ASCII diagram. Assertions: topology regex, agent table header, blast radius levels, ASCII diagram pattern.
3. Phase 3 (Integration): Integration decision matrices per agent. Assertions: service/method/trust table, trust scores.
4. Phase 4 (Confirmation): Per-agent cards with blast radius. Dry run results. Assertions: agent card headers, dry run processed count, blast radius in cards.
5. Phase 5 (Deployment): `.agentbloc/` artifact tree, team.yaml excerpt, governance mention. Assertions: artifact tree, topology in config, governance.yaml.
6. Phase 6 (Evolution): Evolution scan items, patch proposal format, human approval. Assertions: scan items, patch format, human approval mention.

**Assertion patterns (from RESEARCH.md Assertions Strategy, per D-05 and D-06):**
Use these regex patterns for assertion lines, placed AFTER the assistant turn they validate:
- Phase 1: `"pattern": "Phase 1.*Interview.*Gate"`, `"pattern": "(PII|financial|GDPR)"`, `"pattern": "non-technical"`
- Phase 2: `"pattern": "Topology.*Pipeline"`, `"pattern": "\\|\\s*Agent\\s*\\|\\s*Role\\s*\\|"`, `"pattern": "L[1-4]"`, `"pattern": "\\[.*\\].*-->.*\\[.*\\]"`
- Phase 3: `"pattern": "\\|\\s*Service\\s*\\|\\s*Method\\s*\\|"`, `"pattern": "(HIGH|MEDIUM|LOW)"`
- Phase 4: `"pattern": "Invoice Collector"`, `"pattern": "Dry run processed \\d+"`, `"pattern": "Blast Radius.*L[1-4]"`
- Phase 5: `"pattern": "\\.agentbloc/"`, `"pattern": "topology.*pipeline"`, `"pattern": "governance\\.yaml"`
- Phase 6: `"pattern": "(MCP server updates|Security vulnerabilities)"`, `"pattern": "(human approval|no auto-patching)"`

**Critical JSON encoding rules:**
- All content with newlines must use `\n` escape (NOT literal newlines)
- All quotes inside content must be escaped with `\"`
- Pipe characters `|` in markdown tables inside content do not need escaping in JSON
- Backslashes in regex patterns inside the `pattern` field must be double-escaped: `\\s` in the JSON string becomes `\s` when parsed
- Validate every line with `jq . <<< 'line'` mentally -- each line must be independently parseable

**DO NOT:**
- Copy full prose paragraphs from examples (D-03: key decision points, not transcripts)
- Use multi-line JSON (each JSON object must be a single line)
- Forget the state bar in assistant content (SKILL.md mandates it on every response)
- Use overly specific assertion patterns (match structure, not prose -- per RESEARCH.md anti-patterns)
