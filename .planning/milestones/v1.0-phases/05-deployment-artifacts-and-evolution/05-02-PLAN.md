7. **Step 3: Vulnerability Detection (EVOL-03, D-16):** When a CVE is filed against a used MCP server or dependency, generate a "security alert." Structure: CVE ID, severity (cross-ref incident-response.md severity classification), affected agents, affected integration, recommended mitigation (update/replace/disable). Prioritize by severity: P1 (critical) gets immediate Telegram alert, P2-P3 batch into weekly report
8. **Step 4: Patch Proposal Format (EVOL-04, D-17):** Complete patch proposal template with: title, date, scan ID, priority, affected agents table, current state, proposed change, rationale, risk assessment (breaking changes, blast radius impact), rollback plan. The proposal is a structured markdown document delivered via Telegram. User must review before any action
9. **Step 5: Human Approval Gate (EVOL-05, D-18):** Non-negotiable for ALL evolution actions. Flow: proposal sent to Telegram approval thread (per telegram.yaml), user reviews, user replies "approve" or "reject" with optional feedback. Timeout from governance.yaml approvals.default_timeout_minutes. On timeout: do nothing (safe default). On rejection: log reason, do not apply. On approval: apply change, update affected agent configs, log the change to audit trail
10. **Evolution Lifecycle:** Describe the ongoing cycle: scan weekly, accumulate proposals, present batch report unless P1 severity (immediate alert). Track proposal history in state file. Show how the evolution phase transitions from a guided conversation to an automated weekly check
11. **Quick Reference:** Decision tree for the evolution flow: SCAN -> DETECT -> PROPOSE -> APPROVE -> APPLY (or REJECT)

**Critical constraints:**
- Human approval gate is NON-NEGOTIABLE (per D-18). No auto-patches, ever
- Weekly scan frequency is the default but configurable in governance.yaml (per D-14)
- Patch proposals must include rollback plan (per D-17)
- Use Telegram for all approval interactions (per D-18)
- Single `claude -p` session for evolution scans (zero custom runtime, per Research)
