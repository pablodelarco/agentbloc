1. **When This Applies** (2-3 sentences): Claude reads this file during Deployment (Phase 5) when generating governance.yaml audit and rate limiting blocks. Also referenced when configuring Claude Code Hooks for PostToolUse audit logging.

2. **Log Format** (JSONL specification per D-09):
   Each line is a self-contained JSON object. Fields (adapted from IETF Agent Audit Trail draft):
   - `timestamp` (ISO 8601, mandatory): When the action occurred
   - `correlation_id` (string, mandatory): Session identifier format `sess-{agent}-{NNN}` linking related entries
   - `agent` (string, mandatory): Agent name from team.yaml
   - `action` (string, mandatory): What happened (tool_call, decision, error, state_change)
   - `tool` (string, optional): MCP tool or Claude Code tool invoked
   - `target` (string, optional): URL, file path, or resource acted upon
   - `result` (string, mandatory): success / failure / skipped / blocked
   - `duration_ms` (number, optional): Execution time in milliseconds
   - `pii_redacted` (boolean, mandatory): Whether PII redaction was applied to this entry
   - `error` (string, optional): Error message if result is failure

   Example entry (one line in the JSONL file):
   ```json
   {"timestamp":"2026-04-14T10:23:45.123Z","correlation_id":"sess-invoice-collector-001","agent":"invoice-collector","action":"tool_call","tool":"mcp__playwright__navigate","target":"https://provider.example.com/invoices","result":"success","duration_ms":2340,"pii_redacted":true}
   ```

3. **Correlation ID Pattern**:
   - Generated at session start: `sess-{agent_name}-{NNN}` where NNN is a zero-padded sequential number
   - All log entries within a single agent run share the same correlation_id
   - If an agent delegates to a sub-agent, the sub-agent's correlation_id is `{parent_correlation_id}-sub-{NNN}`
   - This allows tracing a complete execution chain across multiple agents

4. **PII Redaction Rules**:
   - ALWAYS redact: person names, email addresses, phone numbers, physical addresses, national IDs, health data, financial account numbers
   - HOW to redact: Replace with `[REDACTED:{field_type}]` in log entries. For data references, use SHA-256 hash of the value, truncated to first 8 characters: `hash:a1b2c3d4`
   - NEVER log: Raw API keys, tokens, passwords, credentials (these should not appear in tool call results)
   - KEEP as-is: Service names, tool names, URLs (stripped of query parameters), file paths, timestamps, status codes

5. **Retention Configuration** (governance.yaml audit block template):
   ```yaml
   audit:
     enabled: true
     format: jsonl
     path: .agentbloc/logs/audit.jsonl
     retention_days: 90
     pii_redaction: true
     correlation_id: true
     fields:
       - timestamp
       - correlation_id
       - agent
       - action
       - result
       - pii_redacted
   ```
   Storage estimate: ~10,000 records/day produces ~15 MB/day. At 90-day retention, ~1.4 GB. For high-volume deployments, consider daily file rotation (audit-YYYY-MM-DD.jsonl) and gzip compression for archived logs.

6. **Rate Limiting Governance** (SECR-06, per D-08 layered approach):
   Two layers: global defaults in governance.yaml and per-agent overrides in agent.yaml.

   governance.yaml rate_limits block:
   ```yaml
   rate_limits:
     global:
       max_cost_usd_daily: 50
       max_api_calls_hourly: 500
       max_tokens_per_session: 100000
     agents:
       invoice-collector:
         max_calls: 100
         period: 1h
         max_cost_usd_daily: 15
       report-sender:
         max_calls: 20
         period: 1h
         max_cost_usd_daily: 5
   ```

   Enforcement: Agent checks remaining budget at session start. If global daily limit reached, agent logs "rate_limit_exceeded" and exits. Per-agent limits checked before each tool call via cron interval spacing.

   Denial-of-wallet protection: If an agent exceeds 80% of its daily cost budget, it sends a Telegram warning. At 100%, it halts and sends an alert.

GDPR patterns activate by default for EU operations. HIPAA and PCI sections activate only when data classification warrants (per D-06).

2. **Art. 17 Right to Erasure** (step-by-step deletion workflow per D-05):
   Generated as part of governance.yaml when GDPR is activated.
   Steps:
   - Receive erasure request: Log DSAR with correlation_id, record timestamp
   - Verify requester identity: Match against known data subjects in state files
   - Enumerate data stores: List all .agentbloc/state/ files containing subject data
   - Delete data: Remove subject records from state files. In audit logs, retain hash-only references (do not delete audit entries -- replace PII with hashes)
   - Confirm deletion: Send confirmation via Telegram to requester thread
   - Notify downstream: If data was shared with external services via MCP, send deletion notification
   - Response deadline: 30 days from request (extendable to 90 days with justification)
   - Exceptions: Legal obligation retention, defense of legal claims

3. **Art. 15 DSAR Workflow** (Data Subject Access Request per D-05):
   - Receive DSAR: Log with correlation_id
   - Verify identity
   - Enumerate all data held about the subject across .agentbloc/state/ files
   - Export format: JSON file containing all data points with processing purposes
   - Response deadline: 30 days
   - Free of charge for first request; reasonable fee for repetitive requests

4. **Art. 33 Breach Notification** (72-hour template per D-05):
   Template fields:
   - Nature of breach (what happened, how many records, what data types)
   - Likely consequences (identity theft risk, financial exposure)
   - Measures taken (kill switch activated, credentials rotated, affected users notified)
   - DPO contact information
   - Timeline: Discovery -> Assessment (immediate) -> Supervisory authority notification (72h) -> Affected individuals notification (without undue delay if high risk)
   - Telegram alert: Immediate P1 alert to operations thread with breach summary

5. **Art. 6 Consent and Legal Basis Logging** (per D-05):
   Each data processing activity logged with:
   - Processing purpose (e.g., "invoice collection for client billing")
   - Legal basis (consent / contract / legal obligation / vital interest / public task / legitimate interest)
   - Data categories processed
   - Retention period
   - Template for governance.yaml:
   ```yaml
   gdpr:
     processing_activities:
       - purpose: "Invoice collection"
         legal_basis: "contract"
         data_categories: ["supplier_name", "invoice_amount", "due_date"]
         retention_days: 365
   ```

6. **DPO Designation Guidance** (per D-05, research Open Question 1 resolution):
   3-question decision tree:
   - Is the data controller a public authority? YES -> DPO required
   - Does core activity involve large-scale, regular, systematic monitoring of individuals? YES -> DPO required
   - Does core activity involve large-scale processing of special categories (Art. 9) or criminal data? YES -> DPO required
   - All NO -> DPO recommended but not mandatory

   DPO designation template (for governance.yaml):
   ```yaml
   gdpr:
     dpo:
       required: true
       name: "[Client's DPO or external DPO service]"
       email: "[dpo@client.example.com]"
       registered_with: "[Supervisory authority name]"
   ```
   Note: AgentBloc generates the template. The client fills in details and validates with legal counsel.

7. **DPA Template** (Data Processing Agreement outline per D-05, for B2B consulting):
   Sections to include in generated DPA outline:
   - Parties (controller = client, processor = consulting entity operating AgentBloc)
   - Subject matter and duration of processing
   - Nature and purpose of processing
   - Types of personal data and categories of data subjects
   - Controller obligations and rights
   - Processor obligations (security measures, sub-processor approval, breach notification, audit rights)
   - Data return/deletion on contract termination
   Note: "AgentBloc generates this outline. Both parties should review with legal counsel before signing."

8. **HIPAA Ready Patterns** (SECR-08, per D-06 -- activated only when PHI detected):
   - PHI safeguards: Encryption at rest (file-level) and in transit (TLS for all MCP connections)
   - Minimum necessary standard: Agents access only the PHI fields required for their specific task
   - BAA requirement: If a third-party MCP server processes PHI, a Business Associate Agreement is required. AgentBloc flags this during Integration Analysis.
   - Audit requirements: All PHI access logged with correlation IDs. Logs retained 6 years minimum.
   - 2025 Security Rule note: All safeguards are mandatory (not addressable). Encryption is required, not optional.

9. **PCI Ready Patterns** (SECR-08, per D-06 -- activated only when financial data detected):
   - Tokenization: Never store raw PAN (Primary Account Number). Use payment provider tokenization (Stripe tokens, payment gateway tokens).
   - If an agent must process card data: restrict to tokenized references only. Raw card numbers never enter .agentbloc/state/ files.
   - PCI DSS 4.0 Requirement 3 summary: Protect stored account data via tokenization or truncation.
   - Agent restriction: Any agent processing financial data is automatically Level 3+ blast radius with requires_approval: true.
