## Division of labor (GSD ↔ GStack)

**GSD owns**:
- `.planning/` memory (ROADMAP.md, PROJECT.md, STATE.md, phases/)
- Milestone + phase lifecycle
- Verification gates
- Requirements traceability
- Spec-driven research workflow
- Spike (`/gsd-spike`) for validating hypotheses before committing
- AI integration phase (`/gsd-ai-integration-phase`) — built for designing AI systems

**GStack contributes INSIDE GSD phases**:
- RESEARCH → `/office-hours` (CEO spec extraction)
- PLAN → `/plan-ceo-review`, `/plan-eng-review`, `/plan-design-review` (multi-perspective)
- EXECUTE → `/autoplan` to refine tasks; gsd-executor runs them
- CODE REVIEW → `/review` (multi-reviewer depth)
- VERIFY → `/qa` (browser), `/cso` (OWASP+STRIDE), `/benchmark`
- RELEASE → `/ship`, `/land-and-deploy`, `/canary` (GSD has no deploy layer)
- COMPLETE → `/document-release`, `/retro`

**Overlap resolutions**:
- Plan: GSD first (PLAN.md), then GStack reviews
- Code review: both sequential (GSD convention + GStack perspective)
- Security: GStack `/cso` primary, GSD just records result
- QA: GSD tests structure, GStack tests experience (complementary)

## Typical v2.0 phase flow (13 steps using both)

```
1. /gsd-discuss-phase N              → CONTEXT.md
2. /office-hours                     → CEO vision doc
3. /gsd-plan-phase N --auto          → PLAN.md with must_haves
4. /plan-ceo-review                  → product-angle review
5. /plan-eng-review                  → engineering review
6. /plan-design-review               → UX review (if applicable)
7. Manual: update PLAN.md with feedback
8. /gsd-execute-phase N              → gsd-executor subagents run
9. /review                           → multi-reviewer pass
10. /qa                              → browser QA (if UI)
11. /cso                             → security audit
12. /gsd-verify-work N               → verify must_haves
13. phase complete                   → state advances
```

Per-phase: 1-3 hours. Per milestone: ~6-10 phases.
